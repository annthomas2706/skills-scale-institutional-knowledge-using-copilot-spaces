# OctoAcme Project Management Docs

Welcome to OctoAcme's project management documentation hub. This README provides a high-level overview of how OctoAcme runs projects and serves as the navigational entry point to all detailed process documents.

---

## Executive Summary

OctoAcme follows a structured, iterative project lifecycle built around five core phases: **Initiation**, **Planning**, **Execution**, **Release**, and **Close & Retrospective**. Every project begins by clearly defining a problem statement, aligning stakeholders, and producing a lightweight Project Charter. From there, the team creates a scoped backlog, maps dependencies, and establishes milestones before moving into delivery. Throughout the lifecycle, the guiding principles remain constant—customer-first thinking, iterative delivery of small testable increments, clear ownership, and data-informed decision-making.

Effective collaboration depends on well-defined roles and a predictable communication rhythm. The **Project Manager (PM)** coordinates delivery, schedules, and risk, while the **Product Manager (PdM)** owns the product vision and backlog prioritization. **Developers** design and build to acceptance criteria, and **QA/Testing** validates quality before release. The team stays aligned through twice-weekly standups, a weekly PM–PdM sync, and monthly stakeholder updates, with ad-hoc escalations handled via a clear path from team-level through PM and Product Lead to the project Sponsor.

Day-to-day execution focuses on delivering working software in short iterations while maintaining transparency. The team tracks progress on a sprint backlog, reviews risks weekly, and keeps a living Risk Register that captures impact, likelihood, ownership, and mitigation plans. Quality is enforced through acceptance criteria, a Definition of Done, code reviews, and automated testing. Blockers and risks are surfaced promptly using standardized weekly status templates, ensuring stakeholders always have a current, single source of truth.

Once a release is ready, OctoAcme follows a standardized deployment process designed to reduce risk and improve observability. After deployment, the team verifies outcomes, communicates the release, and schedules a retrospective to capture learnings. Retrospective action items feed directly back into the planning cycle, creating a continuous improvement loop. This culture of reflection and iteration—grounded in psychological safety—ensures that OctoAcme's processes evolve alongside the team's needs.

## Quick Reference

### Key Roles at a Glance

| Role | Primary Responsibility |
|---|---|
| Project Manager (PM) | Delivery coordination, scheduling, risk management, stakeholder communication |
| Product Manager (PdM) | Product vision, backlog prioritization, success metrics |
| Developer | Feature implementation, testing, code reviews, technical design |
| QA / Testing | Quality validation, acceptance criteria verification |
| Stakeholder | Inputs, approvals, and milestone sign-offs |

### Communication Cadence

| Cadence | Meeting / Event |
|---|---|
| Daily | Standup (delivery team) |
| Twice-weekly | Sprint standup / team sync |
| Weekly | PM + PdM alignment sync; status update to stakeholders |
| Monthly | Stakeholder progress update |
| Per release | Release announcement and retrospective |
| As needed | Ad-hoc risk escalations |

### Essential Artifacts and Templates

| Artifact | Purpose |
|---|---|
| Project Charter / One-pager | Defines scope, goals, and stakeholders at initiation |
| Roadmap and Release Plan | High-level timeline and milestone targets |
| Sprint / Iteration Backlog | Prioritized list of work items for the current sprint |
| Acceptance Criteria & Definition of Done | Quality bar for completing a work item |
| Risk Register | Tracks risks by ID, impact, likelihood, owner, and mitigation |
| Weekly Status Report | Structured update: progress, next steps, risks, and decisions needed |
| Retrospective Notes & Action Items | Captured learnings and assigned improvement actions |
